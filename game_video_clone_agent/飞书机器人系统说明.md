## 一、系统定位

飞书机器人是本项目的"远程总控台"，负责：

1. 选题与立项对话；
2. 剧本审批与重写；
3. 定妆照审批与重画；
4. 触发工业化视频管线（Step 1~3）；
5. 回传进度、结果与错误信息。

```text

  ├─ feishu/session/               ← Session 模型 + SQLite 持久化
  │   ├─ session.py                ← 数据模型 + 状态转移验证

  ├─ feishu/cards/                 ← 卡片子系统
  │   ├─ base_card.py              ← 基类（send/patch/send_or_patch）
  │   ├─ idle_card.py              ← 空闲态（选题盲盒）
  │   ├─ synopsis_card.py          ← 大纲审批卡
  │   ├─ character_card.py         ← 定妆审批卡
  │   ├─ storyboard_card.py        ← 分镜审核卡
  │   ├─ progress_card.py          ← 生产进度卡（运维控制台）
  │   └─ factory.py                ← 卡片工厂（按状态自动选卡）
  ├─ feishu/handlers/              ← 消息与按钮处理器

  │   ├─ message_router.py         ← 消息路由器（关键词 + LLM 兜底）
```

## 三、飞书交互流程（用户视角）

### 1) 启动

- 通过启动脚本拉起 `watchdog.py`；
- `watchdog.py` 确保 `hub.py` 常驻；
- `hub.py` 建立飞书连接并开始接收消息事件。

### 2) 选题与立项

- 用户请求"来一批/换一批" -> 推送选题卡片（IdleCard）；
- 用户发送主题 -> 进入剧本生成。

### 3) 剧本审批

- 生成 `temp_synopsis.json` 后推送大纲卡（SynopsisCard）；
- 卡片上可调时长（±30 秒 / 预设 3/6/10/15 分钟）；
- 用户"通过" -> 进入长文案与定妆照流程；
- 用户"重写" -> 回到大纲重生。

### 4) 定妆照审批

- 生成主角阶段定妆（child/youth/middle/elderly 按剧情裁剪）；
- 推送定妆审批卡（CharacterCard），支持：
  - 全部通过
  - 单阶段重画
  - 配角单独重画
  - 保留剧本重画全部人物
  - 重写剧本 + 全部重画
- 通过后进入全量视频生产。

### 5) 自动生产与回传

机器人串行执行并回传（FIFO 队列 + 原地刷新进度卡 ProgressCard）：

```text
[1/3] Step 1：音频与物理时间轴（step1_writer_v6.py）
       └─ 完成后进入 STEP1_READY
[2/3] Step 2：全量生图（step2_comic_generator_v6.py）
       ├─ Step2-A 宫格生图 → 推送分镜审核卡（StoryboardCard）
       ├─ 用户打回某批 → 单批重画 → 原地刷新审核卡
       ├─ 用户确认全部 → Step2-B 裁切高清
       ├─ 成功 → STEP2_SUCCESS（门禁通过，可进入 Step3）
       └─ 失败 → STEP2_FAILED（阻断合成，需重试 Step2）
[3/3] Step 3：最终合成（step3_assembler_v6.py）
```

## 五、状态机（业务层）

```text
IDLE
  ↓
WAITING_TOPIC
  ↓
GENERATING_SYNOPSIS
  ↓
WAITING_SYNOPSIS_APPROVAL    ← 大纲审批卡（SynopsisCard）
  ↓
GENERATING_VISUALS
  ↓
WAITING_CHARACTER_APPROVAL   ← 定妆审批卡（CharacterCard）
  ↓
STEP1_WRITING                (Step 1: step1_writer_v6.py)
  ↓
STEP1_READY                  (Step 1 完成，等待进入 Step 2)
  ↓
STEP2_GENERATING             (Step 2: step2_comic_generator_v6.py)
  ├─→ WAITING_STORYBOARD_APPROVAL  ← 分镜审核卡（StoryboardCard）
  ├─→ STEP2_SUCCESS                (Step 2 门禁通过，可进入 Step3)
  └─→ STEP2_FAILED                 (Step 2 失败，阻断进入 Step3)
        ↓
      (重试 → STEP2_GENERATING)

STEP2_SUCCESS
  ↓
STEP3_ASSEMBLING             (Step 3: step3_assembler_v6.py)
  ↓
COMPLETED
```

失败与暂停分支：
```text
任意状态 → ERROR → 重试（retry_failed_stage）→ 回到对应审批态或生产态
STEP1_WRITING → PAUSED（暂停态，可恢复）
STEP2_GENERATING → PAUSED（暂停态，可恢复）
STEP3_ASSEMBLING → PAUSED（暂停态，可恢复）
PAUSED → 恢复 → 回到对应生产态（STEP1_WRITING/STEP2_GENERATING/STEP3_ASSEMBLING）
```

状态持久化由 `feishu/session/session_store.py`（SQLite `sessions` 表）负责，重启后可恢复上下文与阶段。状态转移合法性由 `feishu/config.py` 中 `ALLOWED_TRANSITIONS` 定义，`feishu/session/session.py` 中 `Session.transition_to()` 强制执行校验。

## 六、关键可靠性机制（已生效）

1. **快速通道指令**：`可以`、`下一步`、`重启`、`状态` 等高频命令可绕过 LLM，直接执行代码分支（由 `MessageRouter` 和 `config.py` 中 `APPROVE_PHRASES` 等词表驱动）。
2. **定妆照智能裁剪**：按剧情文本自动决定需要哪些人生阶段，减少无效算力消耗。
3. **全局调度队列**：所有重任务统一进入 FIFO 串行队列，避免并发抢占导致飞书与本地双崩。
4. **3秒 ACK 防雪崩**：指令先秒回"已排队"，重任务在 worker 中执行，阻断飞书重试风暴。
5. **精准中止**：按 Run-ID 的 PID 注册表定向终止，替代按进程名全局屠杀。
6. **暂停/恢复**：生产进行中可暂停（PAUSED 状态），后续点按钮恢复。
7. **Step2 失败治理**：
   - `step2_comic_generator_v6.py` 对每个批次最多重试 5 次（生图/解码/切图或高清），由 `STEP2_MAX_BATCH_RETRIES` 控制；
   - 五次失败即中止 Step2，不再补齐占位图续跑；
   - 失败明细写入 `data/runs/<run_id>/logs/step2_failures.jsonl`；
   - Step2 汇总报告写入 `data/runs/<run_id>/logs/step2_report.json`；
   - Hub 在 Step2 子进程非 0 退出或门禁未通过时，推送失败通知并更新进度卡。
8. **Step3 准入门禁**：
   - 进入 Step3 前强制读取 Step2 报告；
   - 若 Step2 未通过或产物不完整，状态置为 `STEP2_FAILED` 并阻断合成。
9. **Step1 Phase3 分镜硬门禁**：
   - 每个 Beat 的 subshot 由两次独立 LLM 调用完成；
   - 单 Beat 内整轮最多重试 5 次（`MAX_VISION_LLM_RETRIES`）；仍失败则进程非 0 退出。
10. **运维控制台卡片（ProgressCard）**：
    - 支持 PATCH 原地刷新进度条（非刷屏新消息）；
    - 显示 Run-ID、资产探针、队列状态、队列预览；
    - 卡片底部按钮支持查看进度、重跑 Step1/2/3、紧急中止、暂停/恢复。
11. **状态转移强制执行**：
    - `ALLOWED_TRANSITIONS` 定义所有合法转移；
    - `Session.transition_to()` 在运行时拦截至非法转移并抛出 `ValueError`。
12. **重启恢复**：
    - Hub 启动时自动检测未完成项目（从 `sessions` 表查询）；
    - 孤儿 Step3 自愈（Step2 已完成但缺成片时自动补跑）；
    - 向最近交互用户推送恢复提示卡。

## 七、运维指令（文本 + 卡片按钮）

- 文本命令：
  - `/status` 或 `状态`：查看当前进度与资产探针
  - `/switch Run_YYYYMMDD_xxx`：切换到历史批次
  - `/retry step1` / `/retry step2` / `/retry step3`：断点续传
  - `换一批`：刷新选题
  - `帮助` / `指令`：查看完整指令手册
- 状态卡按钮动作：
  - `ops_status`：查看最新进度
  - `ops_retry_step1` / `ops_retry_step2` / `ops_retry_step3`：断点续传
  - `ops_abort_run`：紧急中止
  - `pause_project` / `resume_production`：暂停/恢复（新增）

指令纪律：
- 重跑 Step1/2/3 必须入队（不插队）；
- 重跑前做资产校验，不满足条件直接驳回；
- Step3 缺主音轨必须硬拦截。

## 八、运行与排障入口

- 启动入口：`STARTUP_SILENT.bat`
- 主控脚本：`feishu/hub.py`（重构版，~230 行）
- 旧版备份：`feishu/hub_old.py`（可秒回滚）
- 守护脚本：`feishu/watchdog.py`
- 错误上下文：存储在 `sessions` 表的 `error_context` 字段（`SessionStore` 管理）
- 回归测试：`feishu/test_check.py`（104 项检查）
- Step2 结构化日志：`data/runs/<run_id>/logs/step2_failures.jsonl`
- Step2 准入报告：`data/runs/<run_id>/logs/step2_report.json`

建议排障顺序：

1. 先看飞书返回的阶段提示；
2. 再看 `feishu/config.py` 确认状态常量与转移规则；
3. 运行 `python feishu/test_check.py` 做回归检查；
4. 查看 `sessions` 表（SQLite）中的错误上下文；
5. 最后定位当前 Run-ID 下的 `logs/step2_failures.jsonl` 与对应 step 脚本。

### 回滚方法

```bash
# 秒级回滚到旧版 hub.py
cd feishu
del hub.py
ren hub_old.py hub.py
```
