## 一、系统总体架构

```
用户（飞书手机端）
      │
      │  WebSocket 长连接
      ▼
 [飞书开放平台]
      │ 事件推送
      ▼
 feishu/hub.py          ← 核心指挥部（AI大脑+指令路由）
      │
      ├── feishu/state_mgr.py   ← 状态机（SQLite 持久化）
      ├── feishu/ideator.py     ← 选题生成器
      ├── feishu/feishu_mgr.py  ← 飞书消息收发封装
      │
      ├── src/run_story_planner_with_mock.py  ← 剧本规划
      ├── src/ref_generator.py               ← 定妆照生成
      ├── src/step1_writer_v6.py             ← 分镜脚本
      ├── src/step2_comic_generator_v6.py    ← 全量生图+配音
      └── src/step3_assembler_v6.py          ← 视频合成
```

**守护进程：**  
`feishu/watchdog.py` 每 15 秒扫描一次 `hub.py` 是否在运行，挂了就自动拉起，是整个系统的"保安"。

```
双击桌面「启动飞书后台.lnk」

  ├── 杀掉所有残存的 hub.py / watchdog.py 进程

```

### 阶段 1：选题（IDLE → WAITING_TOPIC）

| 步骤 | 你发送 | 机器人回复 | 系统状态变化 |
|------|--------|-----------|-------------|
| 1 | 任意消息（如"来一批"、"给我看看"） | 推送一张含 10 个真实历史/现实职业题材的飞书卡片 | IDLE → WAITING_TOPIC |
| 2（可选） | "换一批" | 重新生成 10 个**排除上一批**的新选题卡片 | 不变 |

> **幕后逻辑：**  
> `ideator.py` 调用千问大模型，强制要求"真实存在的职业"，并把上一批标题传给大模型做去重（`exclude` 参数），确保换一批真的换全新的。

**快速通道关键词（无需经过 AI，直接执行）：**
- 换一批：`换一批`、`刷新`、`换个`、`再来一批`
- 出主意：`盲盒`、`出题`、`给我出个主题`

---

### 阶段 2：立项（→ GENERATING_SYNOPSIS）

| 步骤 | 你发送 | 机器人回复 | 系统状态变化 |
|------|--------|-----------|-------------|
| 1 | 直接发送你想拍的主题（如："我想拍维多利亚时期海盗的一生，7分钟"） | "✅ 立项动作激活：【XXX】..." | WAITING_TOPIC → GENERATING_SYNOPSIS |
| 2 | — | 大模型生成大纲后，推送📖剧情大纲 | → WAITING_SYNOPSIS_APPROVAL |

> **幕后逻辑（立项时的 AI 判断）：**  
> 你的消息进入 AI 导演的大脑（千问模型），系统 Prompt 告诉它当前是"选题阶段"，只有两个权限：`[TRIGGER_START]` 或 `[TRIGGER_IDEAS]`。大模型判断你想立项后，在回复末尾写 `[TRIGGER_START: 主题名, 时长]`，代码解析后自动调用 `run_synopsis_setup()` 生成大纲。

---

### 阶段 3：大纲审批（WAITING_SYNOPSIS_APPROVAL）

| 步骤 | 你发送 | 机器人回复 | 系统状态变化 |
|------|--------|-----------|-------------|
| 1 | 看完大纲，说"可以"/"下一步"/"继续"等 | "✅ 收到！大纲已确认，正在启动定妆照生成..." | → GENERATING_VISUALS |
| 1' | 说"故事不好，结尾太平，改成XXX" | "🔄 已传达给大模型导演，正在重写..." | → GENERATING_SYNOPSIS |
| — | — | 重写完成后，再次推送新大纲 | → WAITING_SYNOPSIS_APPROVAL |

> **幕后逻辑（审批阶段的双层保障）：**  
> **第一层（快速通道）**：`可以`、`下一步`、`继续`、`开始`、`好了` 等 20+ 个词，**完全绕开大模型**，代码直接判断当前状态 = WAITING_SYNOPSIS_APPROVAL → 执行生图。  
> **第二层（AI 理解）**：对于说法更复杂的确认（"就这个吧，不过加长一点结尾"），先走快速通道判断不命中，再交给大模型解析，大模型输出 `[TRIGGER_APPROVE_SYNOPSIS]` 或 `[TRIGGER_REVISE_SYNOPSIS: 具体意见]`。

---

### 阶段 4：定妆照审批（WAITING_CHARACTER_APPROVAL）

| 步骤 | 你发送 | 机器人回复 | 系统状态变化 |
|------|--------|-----------|-------------|
| 1 | 查看定妆照，点卡片上的"✅ 全部通过" | 开始全量视频生产 | → STEP1_WRITING |
| 1' | 文字说"可以"/"下一步" | 同上 | → STEP1_WRITING |
| 1'' | 说"幼年照片不好看，重画" | **在原卡片上**直接刷新幼年图片 | 不变 |
| 1''' | 点卡片"👶 重画幼年" | **在原卡片上**直接刷新幼年图片 | 不变 |
| 1''''| 点卡片"🎨 保留剧本，重画全部人物" | 剧本不变，**在原卡片上**同时刷新全部定妆图 | 不变 |

> **💡 智能阶段裁剪 (Smart Pruning)**：
> 系统会根据剧本大纲的时间跨度，由 AI 自动判断主角经历了哪些阶段。如果剧本没有讲到幼年或老年，卡片上会**明确提示“剧本未涉及该阶段”**，并且会**自动隐藏对应的重画按钮**，防止不必要的误操作和算力浪费。

> **金库实时备份说明**：
> 系统生成这三张图后，在**发送卡片给你之前**，就已经实时将其存入了物理磁盘的 `data/refs/` 和金库备份目录中。你点"✅ 全部通过"时，系统直接从硬盘读取这些图片，无需再额外保存。

### 阶段 5：全量视频生产（STEP1 → STEP2 → STEP3）

这个阶段**全自动运行**，你只需等待。机器人会主动推送进度：

```

```

## 四、上下文记忆机制

### 问题根源
飞书机器人是**异步无状态**的——每收到一条消息启动一个新线程，大模型本身没有跨对话记忆。

### 解决方案：`CHAT_HISTORY` 全局共享记忆库 (已接入 SQLite 持久化)

```python
CHAT_HISTORY = {}  # 内存缓冲，格式: { "open_id": [ {role, content}, ... ] }
MAX_HISTORY_LEN = 20  # 保留最近20轮对话
```

由于之前的内存变量会在重启时丢失，现在系统**已将 `CHAT_HISTORY` 接入 `feishu/state.db` SQLite 数据库**。每次 `hub.py` 重启时，系统会自动从数据库读取最近 20 条对话记录加载到内存，AI 导演再也不会“失忆”。

**两个核心机制：**

1. **用户消息写入**：每条用户消息被 `add_to_history(open_id, "user", msg)` 记录。

2. **后台线程反向写入**（关键！）：  
   系统自动发出去的大纲、定妆照通知，也会调用 `add_to_history(open_id, "assistant", msg)` 写进记忆。  
   这样 AI 导演就"看得到"自己刚给你发了什么大纲，不会再傻乎乎地重新生成。

**每次 LLM 调用的消息结构：**
```
[system_prompt]   ← 当前状态 + 当前权限指令集
[user]   你的第1条消息
[assistant]  机器人回复1
[user]   你的第2条消息
[assistant]  大纲内容（后台线程写入）
[user]   "可以，下一步"    ← 你当前发的这条
```

## 五、动态权限控制（防止 AI 乱触发）

这是解决"AI 在审批大纲时又重新立项"的核心机制：

| 当前状态 | AI 拥有的指令权限 | 被锁死的指令 |
|---------|-----------------|-------------|
| IDLE / WAITING_TOPIC | `TRIGGER_START`、`TRIGGER_IDEAS` | — |
| WAITING_SYNOPSIS_APPROVAL | `TRIGGER_APPROVE_SYNOPSIS`、`TRIGGER_REVISE_SYNOPSIS` | **`TRIGGER_START` 被锁死** |
| WAITING_CHARACTER_APPROVAL | `TRIGGER_APPROVE_CHARACTER`、`TRIGGER_REGEN_CHARACTER` | — |

在大纲审批阶段，系统 Prompt 会明确写：  
> "**绝对禁止**在大纲审批阶段输出 `[TRIGGER_START]`，老板说'可以'绝对是在对当前大纲表态，不是在重新立项！"

## 六、快速通道指令（绕开 LLM，100% 可靠）

以下指令**直接由 Python 代码处理**，不经过 AI 大模型，不受任何 Prompt 约束影响：

| 指令词 | 触发条件 | 执行动作 |
|--------|---------|---------|
| `状态` / `进度` | 任何时候 | 报告当前流程状态 |
| `帮助` / `救命` / `指令` | 任何时候 | 显示操作手册 |
| `重置` / `清空` / `强行重开` | 任何时候 | 强制恢复 IDLE 状态 |
| `重启` / `复活` | 任何时候 | 冷重启 hub.py 进程 |
| `可以` / `行` / `下一步` / `继续` 等 | WAITING_SYNOPSIS_APPROVAL | 批准大纲，进入生图 |
| `可以` / `下一步` 等 | WAITING_CHARACTER_APPROVAL | 批准定妆照，进入生产 |
| `换一批` / `刷新` 等 | 任何时候 | 重新生成选题卡片 |
| `盲盒` / `出题` 等 | 任何时候 | 推送选题卡片 |

## 七、已知限制与注意事项

| 问题 | 原因 | 状态 |
|------|------|------|
| 对话超过 15 轮后早期记忆消失 | `MAX_HISTORY_LEN = 15`，超出会裁剪最早的 | ✅ 可调，改大数字即可 |
| 飞书单文件 ≤50MB 限制 | 飞书平台本身的上传限制 | ⚠️ 超出自动转百度网盘 |
| 进程异常退出 watchdog 才能感知 | watchdog 每 15 秒扫一次 | ✅ 最多延迟 15 秒自愈 |
| 大模型回复语气华丽难以理解 | 千问模型特性 | ✅ 已通过快速通道绕过高频操作 |
| 生成失败时缺乏具体原因 | 早期隐藏了报错日志 | ✅ 已修复，直接将报错输出至飞书 |

## 八、启动与重启操作手册

### 日常启动
```
双击桌面：启动飞书后台.lnk
```

### 手动重启（代码更新后）
**方法一（推荐）**：在飞书直接发送 `重启`  
**方法二**：双击桌面 `启动飞书后台.lnk` （自动触发 `psutil` 清场）  
**方法三**：在项目目录运行 `.\STARTUP_SILENT.bat`  

### 核查进程是否在跑
由于 `wmic` 在 Windows 11 下不稳定，系统已全部换为 `psutil` 模块管理进程。您可以打开任务管理器查找 `python.exe`（命令行包含 `hub.py` 或 `watchdog.py`）。
或在 PowerShell 中快速查看：
```powershell
Get-CimInstance Win32_Process -Filter "CommandLine LIKE '%hub.py%'" | Select-Object ProcessId, CommandLine
```

### 强制清空任务重来
在飞书发送：`强行重开`
