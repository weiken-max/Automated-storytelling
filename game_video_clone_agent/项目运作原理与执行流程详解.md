# 自动讲故事视频项目：运作原理与执行流程详解（V2026.05.08）

> 适用项目：`game_video_clone_agent`  
> 文档目标：以当前代码为准，说明工业化 16 宫格视频生产管线如何稳定运行

---

## 1. 项目核心目标

这是一个“音频先行、时间锁死、画面后配、绝对卡点”的工业化叙事视频系统。

一句话：

1. 先产纯净长文案与定妆照；
2. 先把主音轨和物理时间轴锁死；
3. 再做剧情切分和触发点插值；
4. 最后批量生图并按绝对时间装配成片。

---

## 2. 全链路 Blueprint（当前实现）

### 运行空间隔离（新增）

- 每次任务启动生成唯一 Run-ID：`Run_YYYYMMDD_HHMMSS_xxx`
- 运行目录：`data/runs/<run_id>/...`
- 当前指针：`data/runs/current_run.json`
- Step1/Step2/Step3 只在当前 Run-ID 目录交接数据，杜绝跨批次污染。

### 阶段一：剧本与绝对时钟建立

#### ① 剧本生成与定妆（`src/story_planner_v6.py` + `src/ref_generator.py`）

- 输入：主题（topic）
- 处理：
  - 生成 `temp_synopsis.json`（梗概）
  - 生成 `full_story_v6.json`（1700 字纯净长文案）
  - 并行触发定妆照生成与阶段锚点回写
- 输出：
  - `data/runs/<run_id>/scripts/temp_synopsis.json`
  - `data/runs/<run_id>/scripts/full_story_v6.json`
  - `data/refs/*`（定妆照）

#### ② 音频先行与物理时间轴（`src/step1_writer_v6.py --phase phase2`）

- 输入：`full_story_v6.json` 中 `master_design.full_narration`
- 处理：
  - 调用 Edge-TTS 生成 `master_voice.mp3`
  - 解析 VTT 生成真实微句时间轴 `master_srt.json`
  - 行号锚定后让 LLM 只做 Beat 索引切分
- 输出：
  - `data/runs/<run_id>/audio/master_voice.mp3`
  - `data/runs/<run_id>/audio/master_srt.json`
  - `data/runs/<run_id>/scripts/pseudo_srt.json`

---

### 阶段二：安全隔离的剧情拆解（Beat）

仍由 `src/step1_writer_v6.py --phase phase2` 完成：

- 代码给微句子临时加行号；
- LLM 仅返回 `start_index/end_index`；
- 代码据此回收出带真实时间的 Beat（`pseudo_srt.json`）；
- `pseudo_srt.json.pseudo_srt[*].cues` 为内部必需字段（保存原始微句数组及其物理时间），供 Phase 3 插值算法读取；
- 行号只在中间过程存在，不进入持久产物。

---

### 阶段三：视觉规划与时间插值（Subshot）

#### ③ 分镜与 trigger_time 计算（`src/step1_writer_v6.py --phase phase3`）

- 输入：`pseudo_srt.json` + `full_story_v6.json`（含 `physical_char_anchors` / `stage_map` 等）+ 优先 `scripts/refs-prompt.json`（旧批次可仍用 `character_anchors`）
- 处理（**已解耦为两次独立 LLM 调用，降低单次 JSON 输出过大导致失败概率**）：
  1. **第一次调用**：仅要求模型在 Beat 原文中插入 `<subshot>` 切点；代码将 `tagged_text` 切为 `text_segments[]`，并校验拼接结果与 `beat["text"]` 逐字一致；
  2. **中间层**：对切段做 `_enforce_subshot_segment_density`（长段机械再切等，仍保持与原文拼接一致）；
  3. **第二次调用**：针对最终 `text_segments[]` 逐条生成英文 `visual_prompts[]`；系统 prompt 中附带 **当前 Beat 的 `summary`**、**上一 Beat 的 `summary`（前情）** 以及定妆锚点，便于连戏；
  4. 代码按每个片段首字在 Beat 内 cues 上的字符位置做线性插值，写入 `trigger_time`；
  5. **可靠性**：每个 Beat 上述整轮最多重试 5 次（`MAX_VISION_LLM_RETRIES`）；仍失败则 **`Phase3BeatError` / 进程退出**，并打印 `beat_id`、时间轴区间与失败阶段——**禁止再静默跳过某 Beat**（否则下游缺 subshot，成片会出现某一镜长时间静止）。
- 输出：
  - `data/runs/<run_id>/scripts/narrative_v6_final.json`（此文件产出后，Step1 完成，系统进入 `STEP1_READY` 状态）

该文件核心结构如下：

```json
{
  "metadata": {},
  "timeline": [
    {
      "subshot_id": "S_001",
      "beat_id": "beat_1",
      "trigger_time": 0.0,
      "text_segment": "...",
      "visual_prompt": "...",
      "protagonist_stage": "middle",
      "anchor_look": "data/refs/protagonist_middle/triple_view.png"
    }
  ]
}
```

### 阶段四：16宫格高并发视觉生产

#### ④ 打包生图（`src/step2_comic_generator_v6.py`）

- 输入：`narrative_v6_final.json.timeline`
- 处理：
  - 每 16 条 `visual_prompt` 作为一个批次，要求模型在单张 **16:9** 画幅内生成 `4x4` 宫格；当前对 Gemini 侧请求约 **2560×1440（2K 档位）**，具体以 `image_engine` 中 `imageConfig` 与环境变量为准；
  - 宫格准入由 `src/grid_validators.py` 执行：先做 16:9 与可四等分硬校验，再做“**全图寻峰 + 4×4 结构**”校验（3 条竖线/3 条横线、间距一致性 `GRID_SPACING_MAX_CV`、理论锚点偏差 `GRID_PEAK_ANCHOR_MAX_DEV_RATIO`）；可用 `GRID_VALIDATOR_LEGACY=1` 回退旧规则；
  - 子分镜必须严格按 Prompt 顺序从左到右、从上到下排版（Panel 1 -> Panel 16）；
  - 每批最多重试 5 次（生图 API / 解码 / 切图或高清），由 `STEP2_MAX_BATCH_RETRIES` 控制；单次生图等待默认 **`IMAGE_GEN_TIMEOUT_SECONDS`（默认 180）**，过短会导致服务端其实已出图但客户端已放弃，重复请求浪费配额；
  - 任一批次连续失败 5 次会中止 Step2（不再补占位图）；
  - 生成 `grid_batch_XXX.png`；
  - 调用 `image_processor.py` 将宫格 **像素均分** 为 16 张子镜 `S_XXX.png`；
- 输出：
  - `data/runs/<run_id>/storyboards/grid_batch_XXX.png`
  - `data/runs/<run_id>/storyboards/S_001.png ...`
  - `data/runs/<run_id>/logs/step2_failures.jsonl`
  - `data/runs/<run_id>/logs/step2_report.json`

#### ⑤ 脚本裁切（`src/image_processor.py`）

- 输入：16 宫格大图
- 处理：
  - **仅物理等分**为 4×4（不做 OpenCV 去白边，避免误裁比例）；
  - 每张子图落盘后可选调用 `upscale_image()`：由 `toolsrealesrgan/realesrgan-ncnn-vulkan.exe` 做 Real-ESRGAN 超分并覆盖原 `S_XXX.png`（未找到 exe 或失败时保留裁切原图，不中断流程；可用环境变量 `REALESRGAN_DISABLE=1` 关闭放大，或 `REALESRGAN_MODEL` / `REALESRGAN_SCALE` / `REALESRGAN_GPU` 调参，详见 `src/image_processor.py` 内注释）
- 输出：单镜图 `S_XXX.png`（成功超分后即为放大版）

---

### 阶段五：组装成片

#### ⑥ 绝对时间装配（`src/step3_assembler_v6.py`，纯 FFmpeg）

- 输入：
  - `data/runs/<run_id>/scripts/narrative_v6_final.json`（`trigger_time`）
  - `data/runs/<run_id>/audio/master_voice.mp3`
  - `data/runs/<run_id>/storyboards/S_XXX.png`
- 处理：
  - 进入 Step3 前由 `hub.py` 执行 Step2 准入门禁（读取 `step2_report.json`）；门禁失败时 **飞书侧会收到明确失败提示**（与 Step2 子进程非 0 退出路径一致），避免长时间无反馈；
  - 若 Step2 失败或产物不完整，直接阻断，不执行合成；
  - 每镜时长 = 下一镜 `trigger_time` - 当前镜 `trigger_time`
  - 最后一镜补到音频结尾
  - `duration <= 0` 时自动兜底 `0.1s`
  - 逐镜头生成无声片段 -> concat 清单无损拼接 -> 与主音轨混音导出
  - 临时片段目录自动清理（成功/失败都会清理）
- 输出：
  - `data/runs/<run_id>/output/narrative_v6_final_epic.mp4`

## 3. 当前关键容错机制（已落地）

1. `step1_writer_v6.py`（Phase3）：分镜每 Beat 两次 LLM + 最多 5 轮整 Beat 重试；失败即退出，**不静默丢 Beat**。
2. `step2_comic_generator_v6.py`：按批次最多重试 5 次；失败按阶段分类（IMAGE_GEN_API/IMAGE_DECODE/GRID_CROP/UPSCALE）记录；
3. `step2_comic_generator_v6.py`：五次失败即中止 Step2，不再补齐占位图；
4. `grid_validators.py`：默认“全图寻峰 + 结构校验”降低规则误杀；同时用间距一致性与锚点偏差限制拦截异常宫格，失败统一记为 `GRID_LAYOUT_INVALID`；
5. `feishu/hub.py`：Step3 前读取 `step2_report.json` 做准入门禁；未通过时置 `STEP2_FAILED` 并阻断合成，并 **通知飞书**；
6. `feishu/hub.py`：Step2 子进程异常退出时 **通知飞书**（stderr 摘要）；
7. `step3_assembler_v6.py`：`duration <= 0` 自动兜底，避免渲染异常中断；
8. `image_engine.py`：标准 async/await + Semaphore 限流 + **默认较长单次超时（与 HTTP 客户端一致）** + 重试熔断；
9. `project_vault.py`：软复制容错，缺失文件仅告警，不中断全批次备份；
10. `feishu/hub.py`：全局 FIFO 调度 + 3秒内ACK + Run-ID 精准中止 + 状态卡 PATCH 原地刷新。

## 4. 关键数据接口对齐关系

1. `full_story_v6.json.master_design.full_narration`  
   -> `step1_writer_v6.py phase2` 读取并生成音频与物理时间轴。

2. `pseudo_srt.json.pseudo_srt[*].cues`  
   -> `step1_writer_v6.py phase3` 读取并计算 `trigger_time`。

3. `narrative_v6_final.json.timeline[*].visual_prompt`  
   -> `step2_comic_generator_v6.py` 读取并打包生图。

4. `narrative_v6_final.json.timeline[*].trigger_time`  
   -> `step3_assembler_v6.py` 读取并计算每镜绝对时长。

5. `storyboards/S_XXX.png` 命名序列  
   -> `step3_assembler_v6.py` 以 `S_{i+1:03d}.png` 精准取图。

---

## 5. 一句话业务总结

当前版本已升级为“Run-ID 物理隔离 + 物理时间轴驱动 + 纯 FFmpeg 合成 + 飞书可视化运维调度”，可在高压场景下保持稳定的音画对齐与可恢复生产。
